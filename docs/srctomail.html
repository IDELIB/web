1. プログラム全体構成

SRCTOMAIL は IBMi のオブジェクトとして *PGM / *SRVPGM / *CMD / *PNLGRP の 4 種で構成される。すべてライブラリ MAI に格納される。

SRCTOMAIL (*CMD)
21 パラメータ
↓ 呼び出し
SRCTOMAIL (*PGM)
SRCTOMAIL.c
↓ BNDSRVPGM
SMTPSRVS (*SRVPGM)
SMTPSRVS.c — 27 エクスポート関数
↓ 使用 API
GSKit (gskssl.h)
iconv (QSYSINC)
TCP/IP Socket
DCM API (QSYRGAP)
設計方針：メール組み立てロジック（ヘッダー/ボディ書き込み）を IFS 一時ファイルに蓄積し、最後に SMTPSend() または SMTPSendAuth() で一括送信する二段階方式を採る。これにより本文生成とソケット通信を完全に分離している。

5. SMTP セッション確立フロー

5.1 認証なし (SMTPSend → initSocket)

socket(AF_INET, SOCK_STREAM, 0) — TCP ソケット生成
setsockopt TCP_NODELAY — Nagle アルゴリズム無効化
connect() — SMTPサーバーへ接続（ブロッキングモード）
isSSL > 0 の場合 initGSK() — SSL Over SMTP (Port 465)
recvSocket() — 220 Service Ready 受信
sendSocket("HELO hostname") — HELO 送信
recvSocket() — 250 OK 確認
sendSocket("MAIL FROM:<addr>") — 送信者宣言
sendSocket("RCPT TO:<addr>") × RcptCount — 宛先宣言（ループ）
sendSocket("DATA") — データ転送開始
一時ファイルの全行を sendSocket / sendSocket2 で送信
closeSocket() — <CRLF>.<CRLF> 送信 → "QUIT" → GSKit クローズ

5.2 認証あり (SMTPSendAuth → initSocketAuth)

基本フローは認証なしと同じだが、接続後に EHLO を送信してサーバー機能を取得し、認証処理を挿入する。

socket / setsockopt / fcntl(O_NONBLOCK) — 非ブロッキングで接続（タイムアウト 3 秒）
connect() + select() — 3秒タイムアウト付き非同期接続
fcntl(F_SETFL, flags) — ブロッキングモードに戻す
isSSL==1 → initGSK(GSKAPPID) — SSL Over SMTP (Port 465)
recvSocket() → sendSocket("EHLO hostname") → recvSocket() ループ
250- 行を解析: AUTH PLAIN / LOGIN / CRAM-MD5 / DIGEST-MD5 / STARTTLS を検出
isSMTPS==1 の場合: "STARTTLS" 送信 → 220 確認 → initGSK() → isSSL=1
isSSL==1 の場合: initAuthSSL() — SSL後に再度 EHLO で AUTH を取得
isPLAIN==1 → initAuthPLAIN() / isLOGIN==1 → initAuthLOGIN()
sendSocket("RSET") → 250 確認 — 認証後にセッションリセット
MAIL FROM / RCPT TO / DATA — 以降は認証なしと同じ
注意：CRAM-MD5 / DIGEST-MD5 はサーバー機能として検出されても実装されておらず、最終的に isPLAIN / isLOGIN のどちらかが 1 でなければ DspESCMSG でエスケープメッセージを送出して異常終了する。

7. 半角カタカナ→全角変換 (HanToZen / HanToZen290)

CCSID 5035 向けが HanToZen、CCSID 5026 向けが HanToZen290。それぞれ入力バイト列を走査して次の変換を行う。

  • SO/SI で囲まれた既存 DBCS 区間はそのままコピー
  • SBCS モードで ISKANA マクロが真の文字（半角カタカナ）を検出
  • 次バイトが 濁点（EBCDAKU=190）→ ntbl テーブルで濁点付き DBCS コードに置換
  • 次バイトが 半濁点（EBCMARU=191）→ mtbl テーブルで半濁点付き DBCS コードに置換
  • それ以外 → DBCSPLUS (0x43) + 元コードとしてダブルバイト化
  • カタカナが連続する間は SO...SI ブロックを継続して出力効率を高める
実装補足：c290[128] テーブルは CCSID 1027 から CCSID 290 への単バイト変換テーブル（128 エントリ）。値 -1 はマッピング不可を示す。HanToZen290 はこのテーブルを通じて DBCS コードを決定する。

8. RFC 1522 ヘッダーエンコード

日本語を含むメールヘッダーは RFC 1522 の =?charset?encoding?text?= 形式でエンコードする。実装は RFC1522e() が担う。

char *RFC1522e(char *JISstr) {
  // JIS 文字列を B64Encode でエンコード
  B64strP = B64Encode(JISstr, strlen(JISstr));
  // ISO-2022-JP + Base64 形式でラップ
  sprintf(RFCstr, "=?ISO-2022-JP?B?%s?=", B64strP);
  return RFCstr;
}

件名 (SMTPSubj) の折り返し処理： SO/SI ブロックが 32 文字を超えると強制的に \n （折り返し + スペース）を挿入して新しいブロックを開始する。これは RFC 2047 の 76 文字制限対応の実装である。

フィールド処理関数エンコード対象
FromSMTPFrom → RFC1522eSO/SI で囲まれた DBCS 区間のみ
To/Cc/BccSMTPRcpt → RFC1522e同上
SubjectSMTPSubj → RFC1522eHanToZen 後の全 DBCS 区間（32文字超で折り返し）

9. Base64 エンコードアルゴリズム

B64Encode(char *in, int len) は外部ライブラリを使わず自前実装。標準的な 3 バイト → 4 文字変換を行う。

// 3バイト (c1,c2,c3) → 4文字変換の核心部
ch = c1 >> 2;               B64str[x]   = B64Chars[ch];
ch = ((c1&0x3)<<4)|(c2>>4); B64str[x+1] = B64Chars[ch];
ch = ((c2&0xf)<<2)|(c3>>6); B64str[x+2] = B64Chars[ch];
ch = c3 & 0x3f;             B64str[x+3] = B64Chars[ch];
// 端数処理: 1バイト余り → "xx==" / 2バイト余り → "xxx="
注意：len パラメーターは関数冒頭で len-- されるため、呼び出し時には実際のバイト数をそのまま渡す必要がある（strlen(str) を渡せば正しく動作する）。

添付ファイルの Base64 化では 57 バイト単位（= 76 文字行）で分割して出力する。

// SMTPMpartAttach 内のループ
n = fread(BINstr, 1, 57, fpa);  // 57 bytes → 76 Base64 chars
B64str = B64Encode(BINstr, n);
fprintf(fp, "%s\n", B64str);

11. SMTP 認証フロー (PLAIN / LOGIN)

11.1 PLAIN 認証 (initAuthPLAIN)

認証文字列は \0userid\0password 形式（RFC 4616）。

// BINstr[0] = 0x00 (NUL)
// BINstr[1..usrlen] = ASCII ユーザー名
// BINstr[usrlen+1] = 0x00 (NUL)  ← 実装では BINstr[2+usrlen] に格納
// BINstr[usrlen+2..] = ASCII パスワード
B64str = B64Encode(BINstr, usrlen + pwdlen + 2);
sendSocket("AUTH PLAIN " + B64str);
// 235 2.7.0 Authentication successful を期待
実装上の注意：RFC 4616 は \0authzid\0authcid\0passwd の 3 フィールドだが、本実装は authzid を省略した \0userid\0password の 2 フィールド形式。多くの SMTP サーバーはこの形式を受け入れる。v2.11 では末尾の余分な NUL を除去する修正が入っている。

11.2 LOGIN 認証 (initAuthLOGIN)

sendSocket("AUTH LOGIN") → 334 受信
BINstr = CnvToAsc(AuthUsr) → B64Encode → sendSocket → 334 受信
BINstr = CnvToAsc(AuthPwd) → B64Encode → sendSocket → 235 受信

12. メール本文送信ロジック (sendSocket / sendSocket2)

SMTPSend/SMTPSendAuth では一時ファイルを再オープンし 1 行ずつ読んで送信する。行の種別（ヘッダー / テキストボディ / Base64添付）によって関数を切り替える。

関数対象データ変換処理
sendSocket()SMTP コマンド / ヘッダー行 / Base64 添付データCnvToAsc() で EBCDIC→ASCII 変換後、末尾 CRLF 付加して send()/gsk_write()
sendSocket2()テキストボディ行SO(0x0E) を含む行は Convert()（icd）で CCSID 変換。含まない行は CnvToAsc()。先頭ピリオドの場合は RFC 規定に従い ".." に二重化。改行のみ行は sbyte=0 として CRLF のみ送信
 
// sendSocket: EBCDIC改行(0x0a)除去 → CRLF付加
if (ASCstr[sbyte-1] == 0x0a) { ASCstr[sbyte-1] = 0; sbyte--; }
memcpy(&ASCstr[sbyte], CRLF, 3);  // CRLF = {0x0d, 0x0a, 0x00}
sbyte += 2;

13. MIME マルチパート構造

添付ファイルが 1 件以上の場合、SMTPMIME("multipart/mixed") で boundary を設定し、以下の構造を一時ファイルに出力する。境界文字列は固定値。

MIME-Version: 1.0
Content-Type: multipart/mixed; boundary="4tnD49bUwcnTDkSdRW1Fy0azRXpEwEWURbpEjUSkRI1EkQ8="
Content-Transfer-Encoding: 7bit
X-Mailer: MAI/400 Version 3.00
                              ← 空行 (SMTPBody)
--4tnD49bUwcnTDkSdRW1Fy0azRXpEwEWURbpEjUSkRI1EkQ8=
Content-Type: text/plain; charset=ISO-2022-JP
Content-Transfer-Encoding: 7bit

  [ソースファイル本文 行ごとに SMTPWrite]

--4tnD49bUwcnTDkSdRW1Fy0azRXpEwEWURbpEjUSkRI1EkQ8=
Content-Type: application/octet-stream;
 name="filename"
Content-Transfer-Encoding: base64
Content-Disposition: attachment;
 filename="filename"

  [Base64 エンコード済みデータ 76文字/行]

--4tnD49bUwcnTDkSdRW1Fy0azRXpEwEWURbpEjUSkRI1EkQ8=--  ← SMTPMpartClose
制約：境界文字列は固定値であるため、添付ファイルのバイナリデータ内に偶然この文字列が出現した場合、MIME パーサーが誤動作する可能性がある。実運用上は発生頻度が極めて低いが、理論上の欠陥として認識しておく必要がある。

14. SRCTOMAIL メインプログラム制御フロー

main() は CL コマンドから 21 個のパラメーター（argv[1]〜argv[21]）を受け取り、以下の順に処理する。

argv から全パラメーターをローカル変数にコピー（PTOn, PTO, PFROM, PSERVER, PFILE, PMBR, PSUBJECT 等）
PSSL > 0 の場合 SMTPRegDCM(PGSKAPPID, PSMTPPort) — DCM APPID 登録・SSL/SMTPS モード確定
EdtName() — PFILE(ライブラリ/ファイル名) + PMBR からレコード入力ファイル名文字列を生成
_Ropen(infile, "rr") — IBM i ネイティブ記録 I/O でソースファイルをオープン
_Ropnfbk() — オープンフィードバックからレコード件数・レコード長・isSRCF フラグを取得
SMTPSetDebug / SMTPSetSftSp / SMTPSetConvert(fccsid, tccsid, kana) — サービスプログラム初期設定
SMTPOpen(PSERVER, PSMTPPort) — 一時ファイル作成・iconv 初期化
SMTPFrom / SMTPRcpt(To/Cc/Bcc) / SMTPDate / SMTPSubj / SMTPOther / SMTPMsgID / SMTPMIME / SMTPBody — ヘッダー組み立て
PATTACHn > 0 の場合 SMTPMpartText() — マルチパートのテキストパート開始
_Rreadn ループ — レコードを 1 行ずつ読み込み SMTPWrite でボディに追加（isSRCF=='Y' の場合 buf+12 でソース行先頭シーケンスをスキップ）
添付ファイルループ — SMTPMpartAttach × PATTACHn → SMTPMpartClose
PSMTPAuthUsr=="*NONE" → SMTPSend() / それ以外 → SMTPSendAuth()
SMTPClose() / _Rclose(fp) — クリーンアップ

14.1 ソースファイル vs 通常 PF の判別

isSRCF フラグが 'Y' の場合、ソースファイルのレコードには先頭 12 バイト（シーケンス番号 6 + 更新日 6）があるため buf + 12 から本文を取得する。通常のデータベースファイルは buf + 0 から取得する。

14.2 宛先アドレス検証

To/Cc/Bcc の各アドレスは SMTPValidAddr() で @ 文字の存在を確認する。検証失敗時は DspDIAGMSG で診断メッセージを出力するが、処理は継続する（ESC ではない）。

15. エラーメッセージ処理

メッセージ送信は QMHSNDPM API（msg_id=CPF9898）を使用する。メッセージタイプにより動作が異なる。

関数メッセージタイプ動作
DspSTSMSG()*STATUSステータス表示（DebugSW=1 の場合 *DIAG も送出）。処理継続
DspDIAGMSG()*DIAG診断メッセージ。処理継続
DspESCMSG()*ESCAPE_Rclose(fp) 後 QMHSNDPM → exit(0)。処理停止
DspEXTMSG()*DIAG (CPF9899)_Rclose(fp) 後 QMHSNDPM → exit(0)。処理停止（v2.16追加）
例外ハンドラー：main() 冒頭の #pragma exception_handler(_handler, 0, 0, _C2_MH_ESCAPE, _CTLA_HANDLE, "CPF9898") により、自プログラムが送出した CPF9898 エスケープメッセージを自身でハンドルして正常終了させる設計になっている。これは DspESCMSG のエスケープメッセージを上位呼び出しスタックに伝播させないための処置である。

17. 既知の制約と注意点

項目詳細
タイムゾーン固定SMTPDate() のタイムゾーンオフセットが +0900 にハードコードされている。海外環境では Date ヘッダーが不正確になる
固定 MIME 境界Boundary 文字列が固定値のため、添付データ内に同文字列が含まれる場合に MIME 解析が崩れる可能性がある
グローバル状態全変数がモジュールグローバル。マルチスレッド環境・同一アドレス空間での複数同時実行は非対応
メモリ解放コメントアウトSMTPClose() 内の free() 群がコメントアウトされており、B64str/BINstr/ASCstr/JISstr/RFCstr/EBCstr が解放されない。長時間稼働プロセスで繰り返し使用する場合はメモリリークに注意
recvSocket SSL 版SSL 時の recvSocket は 1 回の gsk_read で最大 1024 バイトを受信。SMTP 応答が分割された場合に対応していない
CRAM-MD5/DIGEST-MD5 未実装EHLO で検出されても使用できず、PLAIN/LOGIN も非対応の場合はエスケープエラーになる
試用版ライセンス期限配布バイナリは 2026年12月31日までの使用期限付き試用版。ChkLicKey() で確認される
