tn5250-server 技術解説書

MCP/tn5250-server/src/index.ts  |  TypeScript MCP Server IBM i / AS/400 v0.1.0

1. 概要

このファイルは IBM i (AS/400) の TN5250 端末エミュレーション を MCP (Model Context Protocol) ツール として AI から操作可能にする Node.js サーバーの全実装です。

内部では TN5250Session.jar という Java プログラムをサブプロセスとして起動し、 標準入力 (stdin) へコマンドを送信、標準出力 (stdout/stderr) から画面テキストを受信するという プロセス間通信（IPC） 方式を採用しています。

4. セッション管理（13〜24行）

14interface Session5250 {
15  process: ChildProcess;   // Java サブプロセスへの参照
16  host: string;
17  port: number;
18  user: string;
19  buffer: string[];        // 出力行のリングバッファ
20  connected: boolean;      // プロセス生存フラグ
21}
23const sessions = new Map<string, Session5250>();
24const MAX_BUFFER_SIZE = 1000;
設計のポイント
  • session_id をキーとした Map でマルチセッションを管理。複数の IBM i ホストへ同時接続可能。
  • buffer は最大 1000 行に制限（splice で古い行を削除）。メモリリークを防ぎながら直近の出力だけを保持する。
  • connected フラグ は Java プロセスの exit イベントで false になる。接続状態を非同期で追跡できる。

7-2. send_keys（174〜235行）

セッションの stdin に send <keys>\n という1行テキストを書き込み、 wait_ms ミリ秒待機後にバッファの内容を返します。 

197session.buffer.length = 0;         // 送信前にバッファをクリア
200session.process.stdin?.write(`send ${keys}\n`);
203await new Promise((resolve) => setTimeout(resolve, wait_ms));
206const output = session.buffer
207  .map(line => line.replace(/[\x00-\x1F\x7F-\x9F]/g, ' '))
208  .join("\n");
制御文字の除去：TN5250 プロトコルは画面属性や色情報をバイト列で埋め込みます。 正規表現 /[\x00-\x1F\x7F-\x9F]/g でこれをスペースに置換し、AI が読みやすいプレーンテキストに変換しています。

キー記法の例

入力例意味

[pf3]ファンクションキー F3
[f1]ファンクションキー F1

7-3. get_screen（238〜297行）

stdin に screen\n を書き込み、500ms 後のバッファを返します。 現在表示されている 5250 画面のテキストを取得するための最もシンプルなツールです。

7-4. get_fields（300〜359行）

stdin に fields\n を送信。画面上の全フィールド（保護フィールド含む）の 位置・内容をテキスト形式で取得します。

7-5. get_inputs（362〜421行）

stdin に inputs\n を送信。入力可能フィールドのみを絞り込んで返します。 フォーム入力の自動化に適しています。

7-6. disconnect_5250（424〜472行）

445session.process.stdin?.write("quit\n"); // Java側に正常終了を指示
448await new Promise((resolve) => setTimeout(resolve, 500));
449session.process.kill();                 // 念のため強制終了

quit コマンドで Java プロセスに正常終了を促し、500ms 後に kill() で強制終了します。 sessions.delete() でマップからエントリを除去しメモリを解放します。

StdioServerTransport を使用するため、MCP ホスト（Bob など）が このプロセスをサブプロセスとして起動し、stdin/stdout で JSON-RPC 通信します。 ログは stderr に出力するので、JSON-RPC ストリームと混在しません。

9. データフロー全体像

connect_5250 → send_keys → get_screen の典型的な流れ
  1. AI が connect_5250 を呼び出す → Node.js が java -jar TN5250Session.jar ... を起動
  2. Java が IBM i ホストへ TN5250/Telnet 接続し、ログイン処理を行う
  3. Java の stdout → Node.js の buffer 配列に行単位で蓄積
  4. 3秒後、プロセスが生存していれば接続成功をAIに返す
  5. AI が send_keys で WRKSYSSTS[enter] を送信
  6. Node.js が send WRKSYSSTS[enter]\n を Java の stdin へ書き込む
  7. Java がキーを 5250 プロトコルで IBM i へ送信し、画面データを stdout へ出力
  8. Node.js が 1000ms 待機後、buffer を制御文字除去してAIに返す
  9. AI が情報を解析し、次のアクションを決定する

10. エラーハンドリング戦略

状況対応
