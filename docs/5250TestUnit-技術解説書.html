1. 概要とアーキテクチャ

5250TestUnit は IBM i（AS/400）上で動作する 5250 端末アプリケーションに対して、 画面操作テストケースを自動生成・実行するCLIツールです。 TN5250 プロトコルで実機接続し、実際の画面テキストをアサートするため、 エミュレータ上での目視確認を不要にします。

コンポーネント図

┌─────────────────────────────────────────────────────────────────┐ │ 5250TestUnit CLI │ │ node build/index.js analyze | generate | run │ └──────────────┬──────────────────────────────┬───────────────────┘ │ │ ┌───────────▼───────────┐ ┌─────────────▼────────────────┐ │ SSH Client │ │ SessionDriver │ │ (node-pty / ssh) │ │ (TypeScript class) │ │ analyze / generate │ │ stdin/stdout パイプ通信 │ └───────────────────────┘ └─────────────┬────────────────┘ │ spawn ┌─────────────▼────────────────┐ │ TN5250Session.jar │ │ (Java / HACL ECL API) │ │ ・TN5250 接続 / サインオン │ │ ・画面取得 (screen) │ │ ・キー送信 (send) │ └─────────────┬────────────────┘ │ TN5250 ┌─────────────▼────────────────┐ │ IBM i 7.5 │ │ （RPGプログラム群） │ └──────────────────────────────┘ DbVerifier.jar (JTOpen JDBC) ←→ IBM i DB2 for i

処理フロー（run コマンド）

1
testcases.json を読み込む（TestSuite 型）
↓
2
SessionDriver.connect() → TN5250Session.jar を子プロセス起動 → サインオン
↓
3
テストケースを順番に実行（fail-fast なし）
↓
4
各ステップを実行（keys / assertText / snapshot 等）
↓
5
DB アサーション実行（DbVerifier.jar 経由）
↓
6
JUnit XML + テキストレポートを出力

3. CLIコマンド仕様

コマンド体系は analyze / generate / run の3つです。

analyze — ソース解析

SSH でIBM iに接続し、DSPFソース・RPGソースを取得して解析。{pgm}-analysis.json を出力します。

node build/index.js analyze \
  --host <IBM_I_HOST> \
  --lib  <LIBRARY> \
  --pgm  <PROGRAM> \
  --user <USER> \
  --password <PASSWORD>
オプション必須説明
--host必須IBM i ホスト名または IP
--lib必須ソースライブラリ名
--pgm必須RPGプログラム名
--user必須SSHユーザー名
--password必須SSHパスワード
--port省略可SSHポート（デフォルト: 22）
--dspf省略可DSPFメンバー名（省略時: pgm名）
--rpg省略可RPGメンバー名（省略時: pgm名）
--output省略可出力ファイルパス

generate — テストケース生成

analysis.json と DBサンプリング結果から testcases.json を自動生成します。

node build/index.js generate \
  --analysis <program>-analysis.json \
  --host     <IBM_I_HOST> \
  --user     <USER> \
  --password <PASSWORD> \
  --sample   5
オプション必須説明
--analysis必須analysis.json のパス
--host必須IBM i ホスト名
--user必須SSHユーザー名
--password必須SSHパスワード
--sample省略可取得サンプル件数（デフォルト: 3）
--output省略可出力ファイルパス
--suite-name省略可テストスイート名

run — テスト実行

テストケース JSON を読み込み、TN5250 で実機接続してテストを実行します。

node build/index.js run \
  --testcases <program>-testcases.json \
  --password  <PASSWORD> \
  --code-page 1399
オプション必須説明
--testcases必須testcases.json のパス
--password必須TN5250 パスワード
--host省略可ホスト名（JSON の値を上書き）
--user省略可ユーザー名（JSON の値を上書き）
--port省略可TN5250ポート（デフォルト: 23）
--code-page省略可EBCDICコードページ（デフォルト: 930）
--output-xml省略可JUnit XML 出力パス
--output-txt省略可テキストレポート出力パス
コードページの選択： 日本語環境では通常 930（IBM EBCDIC日本語）または 1399（Unicode対応日本語）を使用します。 接続先 IBM i のシステム値 QCCSID に合わせて指定してください。

4. テストケース JSON スキーマ

テストケースは JSON ファイルで記述します。以下が完全なスキーマです。

{
  "suiteName": "テストスイート名（レポートに表示）",
  "host":      "<IBM_I_HOST>",      // IBM i ホスト名
  "port":      23,                  // TN5250 ポート
  "user":      "<USER>",            // ユーザー名
  "codePage":  "1399",              // コードページ（文字列でも可）
  "tests": [
    {
      "name": "TC001: テストケース名",
      "steps": [ /* TestStep の配列 */ ],
      "dbAssertions": [ /* DbAssertion の配列（省略可）*/ ]
    }
  ]
}

TestStep 型一覧

typeパラメータ説明
keys keys, waitMs キーストロークを送信する。[enter], [pf3] 等の特殊キーが使用可能。送信後プロンプトが返るまで自動待機。
assertText text 現在の画面テキストに指定文字列が含まれるか検証する。失敗時は画面ダンプ（先頭500文字）をメッセージに含む。
assertTitle text 画面1行目（タイトル行）に文字列が含まれるか検証する。
assertField fieldIndex, fieldValue 入力フィールドの値を検証する。fieldIndex は 1 始まりの番号。
snapshot snapshotPath 現在の画面をコンソールにリアルタイム出力し、テキストレポートにも記録する。
waitScreen waitMs 指定ミリ秒間待機する。画面遷移の安定待ちに使用。

特殊キー記法

記法説明
[enter]Enterキー
[pf1]〜[pf24]ファンクションキー F1〜F24
[f1]〜[f24]同上（自動変換される）
[tab]タブキー
[backtab]バックタブ
[clear]クリアキー
[home]ホームキー
[pageup], [pagedn]ページアップ/ダウン
[eraseeof]フィールド末尾まで消去
[reset]エラーリセット

DbAssertion 型

{
  "description": "アサーション説明",
  "sql": "SELECT COUNT(*) AS CNT FROM QEOL/HINMSL01 WHERE HNNBAN = '40100'",
  "expectedRows": 1,
  "expectedValues": [
    { "CNT": "1" }    // 省略可
  ]
}

5. SessionDriver — 5250操作エンジン

src/runner/session-driver.ts に実装されたクラスで、 TN5250Session.jar を Node.js 子プロセスとして起動し、 stdin/stdout パイプ経由で制御します。

プロンプト同期方式

TN5250Session.jar はコマンドごとに画面テキストを出力した後 > プロンプトを出力します。 SessionDriver はこのプロンプトを区切りとして使用し、応答が完全に届くまでポーリングで待機します。

旧実装の問題（解決済み）： 固定 500ms 待機では画面テキスト（80列×24行）が受信完了する前にバッファを読んでいたため、 assertText が常に失敗していました。プロンプト同期方式に変更することで解決しました。

主要メソッド

メソッド説明
connect(host, port, user, password, codePage)JARを起動してサインオン。> プロンプトが出るまで最大30秒待機。
disconnect()quitコマンド送信後にプロセスをkill。
sendKeys(keys, waitMs?)JAR に send {keys} を送信。プロンプト到着後、waitMs で追加待機。
getScreen()JAR に screen を送信し、プロンプト前のテキストを返す。
getFields()JAR に fields を送信し、フィールド情報を返す。
assertText(expected)getScreen() を呼び出し、テキストの存在を検証。
snapshot(label)getScreen() を呼び出し、ラベル付き画面テキストを返す。

制御文字フィルタ（stripCtrl）

画面テキストから 5250 属性バイトなどの制御文字を除去します。保持するコードポイント：

  • 0x0A（LF）、0x0D（CR）— 改行を保持
  • 0x20（スペース）
  • 0x21〜0x7E（印字可能 ASCII）
  • 0xA0 以上（UTF-8 日本語文字を含むマルチバイト）

タイムアウト定数

定数値用途
CONNECT_TIMEOUT_MS30,000 ms接続確立（サインオン含む）の最大待機時間
PROMPT_TIMEOUT_MS15,000 ms各コマンド応答の最大待機時間

6. TN5250Session.jar — Java側プロセス

src/main/java/TN5250Session.java で実装され、 IBM Host Access Class Library (HACL) の ECL API を使用して TN5250 セッションを管理します。

起動引数

java -jar TN5250Session.jar <host> <port> <user> <password> [codePage]

対話モードコマンド

コマンド説明
screen (または sc)現在の画面を80列×24行でテキスト出力
send <keys> (または s)キーを送信し、画面更新完了後に画面テキストを出力
fields (または f)全フィールド情報を出力（型・位置・内容）
inputs (または i)入力可能フィールドのみ出力
helpコマンド一覧を表示
quit (または q)切断して終了

画面テキスト出力形式

screen コマンドは以下の形式で画面を出力します。

================================================================================
  IPL010                  品目マスター照会                     日付  26/07/15
                                                               時刻  20:04:27
    品目番号・・・・・・
  ...（24行）...
================================================================================
>

行頭の ================ 区切り線と末尾の > プロンプトが応答の区切りです。

ビルド

MCP/ ディレクトリで Maven ビルドを実行します。

cd MCP
mvn package

生成先: tn5250-server/lib/TN5250Session.jar（fat JAR、約18MB）

7. DbVerifier — DB事後検証

src/runner/db-verifier.ts が lib/DbVerifier.jar を子プロセスとして起動します。 JSON を stdin に送り、結果 JSON を stdout から受け取ります。

通信プロトコル

入力 JSON（stdin）

{
  "host":     "<IBM_I_HOST>",
  "user":     "<USER>",
  "password": "<PASSWORD>",
  "sql":      "SELECT COUNT(*) AS CNT FROM <LIBRARY>/<TABLE> WHERE <KEY_FIELD> = '<VALUE>'"
}

出力 JSON（stdout）

{
  "rows":     [ { "CNT": "1" } ],
  "rowCount": 1,
  "error":    null            // エラー時は文字列
}

検証ロジック

  1. DbVerifier.jar が存在しない場合 → スキップ（pass=true）
  2. JAR 実行エラー → pass=false
  3. rowCount が expectedRows と不一致 → pass=false
  4. expectedValues が指定されていれば各列の値を比較
  5. 全チェック通過 → pass=true

ビルド

mvn package により 5250testunit/lib/DbVerifier.jar が生成されます（JTOpen内包、約17MB）。

9. analyze コマンド詳細

SSH 接続で IBM i からソースを取得し、解析結果を JSON 出力します。

処理ステップ

1
SSH接続（node-pty）
↓
2
DSPF ソース取得（CPYTOSTMF コマンド → IFS ファイル読み取り）
↓
3
dspf-parser.ts でフォーマット定義・フィールド定義を解析
↓
4
RPG ソース取得 → rpg-parser.ts で参照ファイル一覧を抽出
↓
5
DISK デバイスファイルを db-inspector.ts で検査（カラム定義取得）
↓
6
{pgm}-analysis.json を出力

analysis.json 出力形式

{
  "program": "IPL020",
  "library": "QEOL",
  "dspf": {
    "member": "IPL020D",
    "formats": [
      { "name": "PANEL01", "isSubfile": false, "fields": [...] },
      { "name": "CTL01",   "isSubfile": false, "sflCtlName": "SFL01", "fields": [...] }
    ]
  },
  "rpgFiles": [
    { "fileName": "HINMSL01", "fileType": "I", "device": "DISK", "library": "QEOL" }
  ],
  "dbTables": [
    { "name": "HINMSL01", "library": "QEOL", "fields": [...] }
  ]
}

10. generate コマンド詳細

analysis.json と DB サンプルデータから testcases.json を自動生成します。

生成されるテストケースの種類

  • 初期表示テスト — プログラム起動後の画面タイトルを検証
  • データ検索テスト — サンプルデータのキー値で入力・結果確認
  • DB アサーション — 画面に表示されたデータが DB と一致するか検証
注意：自動生成されたテストケースは出発点です。 実際の画面遷移（メッセージ画面の閉じ方、特殊操作等）は手動で調整してください。

12. テスト作成ガイド

基本パターン：プログラム呼び出しと画面アサート

{
  "name": "TC001: 基本照会テスト",
  "steps": [
    { "type": "waitScreen", "waitMs": 3000 },
    { "type": "keys", "keys": "[enter]", "waitMs": 2000 },
    { "type": "keys", "keys": "ADDLIBLE LIB(QEOL)[enter]", "waitMs": 2000 },
    { "type": "keys", "keys": "CALL PGM(QEOL/IPL010)[enter]", "waitMs": 8000 },
    { "type": "assertText", "text": "品目マスター照会" },
    { "type": "snapshot", "snapshotPath": "初期画面" },
    { "type": "keys", "keys": "40100[enter]", "waitMs": 5000 },
    { "type": "assertText", "text": "21,700" },
    { "type": "snapshot", "snapshotPath": "詳細画面" },
    { "type": "keys", "keys": "[pf3]", "waitMs": 3000 }
  ],
  "dbAssertions": []
}

注意事項

接続直後のメッセージ画面： IBM i では接続時に「メッセージ待ち行列が割り振られている」等の通知画面が表示される場合があります。 TC001 の先頭に { "type": "keys", "keys": "[enter]", "waitMs": 2000 } を 追加して画面を閉じてください。TC002 以降は同一セッションのため不要です。
ADDLIBLE は TC001 のみ： ADDLIBLE LIB(QEOL) は1テストスイートの最初のテストケースのみで実行します。 同一セッション内では持続するため、TC002 以降は不要です（重複しても問題ありませんが効率が下がります）。

waitMs の目安

操作推奨 waitMs
接続直後 waitScreen3,000 ms
ADDLIBLE2,000 ms
CALL PGM（プログラム起動）8,000 ms
Enter（画面内遷移）5,000 ms
F3（終了・戻り）3,000 ms

keys ステップの waitMs はプロンプト到着後の追加待機です。 プロンプト到着まではタイムアウト（最大15秒）まで自動待機するため、 waitMs を省略しても多くの場合は動作します。

13. トラブルシューティング

症状原因対処
assertText が常に失敗する 接続直後のメッセージ画面が表示されたまま操作している TC001 先頭に [enter] ステップを追加
CPF4101: ファイル OPEN エラー 必要なライブラリがライブラリーリストにない ADDLIBLE LIB(QEOL) が実行されているか確認
TN5250Session.jar not found JAR が生成されていない cd MCP && mvn package を実行
Failed to connect: Java process exited ホスト名・ポート・認証情報の誤り、またはサーバーが停止 接続情報を確認し、ネットワーク疎通を確認
画面テキストが文字化けする コードページの不一致 --code-page で正しいコードページを指定（930 または 1399）
画面に保護域エラーが表示される 入力フィールド以外に文字を送信しようとしている 画面状態を確認し、[reset] キーを送信してから再試行
テスト実行時間が長い waitMs が大きすぎる プロンプト同期待機で自動最適化されるため waitMs を 0 に設定可能

デバッグ方法

assertText 失敗時のエラーメッセージには画面ダンプ（先頭500文字）が含まれます。 テキストレポートファイル（*-results.txt）の失敗詳細セクションを確認してください。

また snapshot ステップをテストケースに挿入すると、 任意のタイミングの画面をコンソールとレポートファイルの両方に記録できます。
